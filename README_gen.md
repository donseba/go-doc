# Experimental generated helper namespaces

`@gen` is an experimental go-doc feature for package-like helper namespaces in
ordinary Go templates.

It lets a template say:

```gotemplate
{{/*
@model Page github.com/example/app.Page
@gen time github.com/example/app/internal/timefuncs
@gen money github.com/example/app/internal/moneyfuncs
@gen text github.com/example/app/internal/textfuncs
*/}}

{{ time.Format Page.GeneratedAt "15:04:05" }}
{{ money.EUR .MonthlyCents }}
{{ text.Initials .Owner }}
```

At runtime, `time`, `money`, and `text` are not magical imports. They are normal
template functions registered in a normal `html/template.FuncMap`. Each
generated function returns a small namespace value whose exported methods
forward to exported Go functions.

The `Page` model in the example is a separate two-way contract. `@model Page`
is the template-side entrance. Your application still has to provide the
runtime exit by registering a real template function or accessor named `Page`,
usually through go-doc's renderer. `@gen` does not create `Page`; it only
creates helper namespaces such as `time`, `money`, and `text`. If you render
with direct `tmpl.Execute(w, page)`, use `@dot` and `{{ .Title }}` instead.

That is the whole trick.

## Why this exists

Go templates are deliberately small. They do not import packages, and a running
Go binary cannot call arbitrary packages that were not compiled into it.

This means an annotation such as this cannot work by itself:

```gotemplate
{{/* @import view github.com/example/app/viewfuncs */}}
```

There is no runtime hook where `html/template` can dynamically import that
package. Go needs the package to be imported by compiled Go code.

`@gen` keeps that boundary honest:

1. The template declares the desired helper namespace.
2. `go-doc exp/gen` scans the declaration.
3. The generator writes ordinary Go code.
4. The application imports that generated package.
5. The application registers the generated `FuncMap`.
6. The template calls namespace-style helpers.

So the result feels package-like in the template, while the implementation stays
inside normal Go rules.

## Why the annotation is `@gen`

`@gen` is intentionally not called `@import`.

`@import` suggests the template can import a Go package at runtime. It cannot.
`@gen` says the important part out loud: generated Go code is required.

If this experiment graduates later, the name can still change. For now, `@gen`
is honest about the mechanism.

## Runtime shape

Given these helper packages:

```go
package timefuncs

import "time"

func Format(t time.Time, layout string) string {
    return t.Format(layout)
}
```

```go
package moneyfuncs

import "fmt"

func EUR(cents int) string {
    return fmt.Sprintf("EUR %.2f", float64(cents)/100)
}
```

```go
package textfuncs

import "strings"

func Initials(name string) string {
    // ...
}
```

And these template declarations:

```gotemplate
{{/*
@gen time github.com/example/app/internal/timefuncs
@gen money github.com/example/app/internal/moneyfuncs
@gen text github.com/example/app/internal/textfuncs
*/}}
```

The helper packages stay where normal application code belongs, often under
`internal/`. The generated bridge can be a small package such as `gen`:

```go
// Code generated by go-doc exp/gen; DO NOT EDIT.
package gen

import (
    moneyfuncs "github.com/example/app/internal/moneyfuncs"
    textfuncs "github.com/example/app/internal/textfuncs"
    timefuncs "github.com/example/app/internal/timefuncs"
    template "html/template"
    time "time"
)

type TimeNamespace struct{}
type MoneyNamespace struct{}
type TextNamespace struct{}

func FuncMap() template.FuncMap {
    return template.FuncMap{
        "time":  func() TimeNamespace { return TimeNamespace{} },
        "money": func() MoneyNamespace { return MoneyNamespace{} },
        "text":  func() TextNamespace { return TextNamespace{} },
    }
}

func (TimeNamespace) Format(v0 time.Time, v1 string) string {
    return timefuncs.Format(v0, v1)
}

func (MoneyNamespace) EUR(v0 int) string {
    return moneyfuncs.EUR(v0)
}

func (TextNamespace) Initials(v0 string) string {
    return textfuncs.Initials(v0)
}
```

The app then wires this into the template just like any other helper map:

```go
tmpl := template.New("page.gohtml").Funcs(gen.FuncMap())
```

This registers only the generated helper namespaces. Named models such as
`Page` still need the renderer or equivalent application glue to register the
matching template function before parsing.

The template can call:

```gotemplate
{{ time.Format Page.GeneratedAt "15:04:05" }}
{{ money.EUR .MonthlyCents }}
{{ text.Initials .Owner }}
```

`time`, `money`, and `text` each call a generated template function. Each
function returns its own namespace value. Exported functions from the source
package become exported methods on that namespace value. `html/template` can
call those methods normally.

Multiple `@gen` declarations therefore do not create multiple FuncMaps. They
create one generated package with one `FuncMap()` that contains multiple
namespace entries.

## Multiple declarations

Every `@gen` line declares one namespace:

```gotemplate
{{/*
@gen time github.com/example/app/internal/timefuncs
@gen money github.com/example/app/internal/moneyfuncs
@gen text github.com/example/app/internal/textfuncs
*/}}
```

The generator groups all declarations from the scanned templates and emits one
generated helper package:

```go
func FuncMap() template.FuncMap {
    return template.FuncMap{
        "time":  func() TimeNamespace { return TimeNamespace{} },
        "money": func() MoneyNamespace { return MoneyNamespace{} },
        "text":  func() TextNamespace { return TextNamespace{} },
    }
}
```

Each namespace gets its own generated type:

```go
type TimeNamespace struct{}
type MoneyNamespace struct{}
type TextNamespace struct{}
```

Exported functions from the matching package become methods on that matching
namespace only:

```go
func (TimeNamespace) Format(v0 time.Time, v1 string) string {
    return timefuncs.Format(v0, v1)
}

func (MoneyNamespace) EUR(v0 int) string {
    return moneyfuncs.EUR(v0)
}

func (TextNamespace) Initials(v0 string) string {
    return textfuncs.Initials(v0)
}
```

So the template can use all three namespaces side by side:

```gotemplate
{{ time.Format Page.GeneratedAt "15:04:05" }}
{{ money.EUR .MonthlyCents }}
{{ text.Initials .Owner }}
```

The LSP follows the same shape. It creates one synthetic type per namespace:

```text
$go-doc/gen.time
$go-doc/gen.money
$go-doc/gen.text
```

That is why `time.`, `money.`, and `text.` each complete different methods.
They are separate synthetic types, backed by separate Go packages, registered by
one generated FuncMap.

## Why the editor understands it

The language server treats every `@gen name package/path` line as a generated
namespace contract.

In the index, it projects the namespace into the template model space:

```json
{
  "models": {
    "time": "$go-doc/gen.time",
    "money": "$go-doc/gen.money",
    "text": "$go-doc/gen.text"
  },
  "gens": {
    "time": "github.com/example/app/internal/timefuncs",
    "money": "github.com/example/app/internal/moneyfuncs",
    "text": "github.com/example/app/internal/textfuncs"
  }
}
```

Each synthetic `$go-doc/gen.*` type contains methods copied from the exported
functions in its source package. That gives the editor enough information for:

- completion after `time.`, `money.`, or `text.`
- hover on `money.EUR`
- go-to-definition from `time.Format` to the original function
- diagnostics for wrong argument counts or wrong argument types
- semantic highlighting for the namespace and methods

The namespace itself, such as `money`, points to the helper package source file.
The method, such as `EUR`, points to the actual source function.

## Generation command

Install the experimental generator first:

```sh
go install github.com/donseba/go-doc/cmd/godoc-exp-gen@latest
```

From a module root:

```sh
godoc-exp-gen \
  -package gen \
  -out gen/gen.go
```

When `-templates` is omitted, the generator finds the nearest `go.mod`, walks
that module, and scans every `.gohtml`, `.tmpl`, and `.html` file for `@gen`
declarations. It skips noisy/generated dependency folders such as `vendor`,
`node_modules`, `.git`, `dist`, and `build`.

Inside this repository, the example uses the same installed command:

```sh
godoc-exp-gen \
  -package gen \
  -out gen/gen.go
```

You can still pass explicit template files when you want a narrow generation
step:

```sh
godoc-exp-gen \
  -templates templates/page.gohtml,templates/card.gohtml \
  -package gen \
  -out gen/gen.go
```

Duplicate declarations are allowed when they agree:

```gotemplate
{{/* @gen view github.com/example/app/viewfuncs */}}
```

If one template declares `view` as one package and another template declares
`view` as a different package, generation fails. That protects the runtime
`FuncMap` from ambiguous names.

## Direct generation without templates

For experiments, the generator can wrap a package directly:

```sh
godoc-exp-gen \
  -pkg github.com/example/app/viewfuncs \
  -namespace view \
  -package gen \
  -out gen/gen.go
```

Template-driven generation is preferred because the template remains the source
of truth for what it expects.

## Relationship to `@func`

Use `@func` for individual helpers:

```gotemplate
{{/*
@func money github.com/example/app/viewfuncs.Money
*/}}

{{ money .MonthlyCents }}
```

Use `@gen` when a package naturally belongs together as a namespace:

```gotemplate
{{/*
@gen view github.com/example/app/viewfuncs
*/}}

{{ view.Money .MonthlyCents }}
{{ view.Initials .Owner }}
{{ view.FormatTime Page.GeneratedAt "15:04:05" }}
```

`@func` is explicit and stable. `@gen` is broader and more opinionated. It is
useful when a team already has a small helper package and wants to expose it to
templates without listing every function in every template.

## Relationship to configured functions

Global functions in `.go-doc/config.json` describe helpers that already exist in
the application's FuncMap:

```json
{
  "functions": {
    "money": "github.com/example/app/viewfuncs.Money"
  }
}
```

That configuration does not create runtime helpers. It only tells go-doc that
the helpers are available.

`@gen` is different because it produces Go code that can be registered at
runtime.

## Current limits

`@gen` is still under `exp/`, so treat it as promising but not final.

Current behavior:

- wraps exported package-level functions
- skips generic functions
- supports normal parameters and return values that `html/template` can call
- preserves imports needed by function signatures
- exposes one `FuncMap()` from the generated package
- supports editor completion, hover, diagnostics, semantic tokens, and
  go-to-definition through the LSP

Current non-goals:

- no dynamic runtime imports
- no `unsafe`
- no package constants
- no generated constructors for exported structs
- no automatic runtime registration unless the application imports and calls the
  generated package
- no promise yet that `@gen` is part of the stable v1 contract

## Why this is safe enough

The generated code is ordinary Go. It imports the helper package explicitly.
The compiler type-checks the wrapper. The app decides whether to register the
generated `FuncMap`.

There is no reflection-based dynamic dispatch in the template call path. Once
generated, calls like `view.Money` are normal method calls on a small namespace
value.

That keeps the experiment understandable:

- templates declare intent
- code generation creates the bridge
- Go compilation verifies the bridge
- go-doc gives the editor the same shape

## Example

See [examples/exp-gen](./examples/exp-gen) for a working app.

It demonstrates:

- `@model Page ...`
- `@gen time ...`, `@gen money ...`, and `@gen text ...`
- source helpers in `internal/timefuncs`, `internal/moneyfuncs`, and
  `internal/textfuncs`
- generated helpers in `gen/gen.go`
- runtime registration of helper namespaces with `gen.FuncMap()`
- runtime registration of the named `Page` model through the renderer or
  equivalent application glue
- editor completion and navigation for `time.Format`, `money.EUR`,
  `text.Initials`, and `money.Add`
